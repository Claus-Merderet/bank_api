# Bank API — полное описание для дизайн-сессии

Этот документ самодостаточен: в нём всё, что нужно для создания дизайн-макетов фронтенда. Каждое тело ответа ниже снято с живого бэкенда (2026-07-04), а не из встроенной документации. Реальные JWT-токены и хэши паролей в примерах заменены плейсхолдерами `<JWT>` и `<bcrypt-hash>`.

---

## 1. Что это за система

Учебный банковский API-полигон, на котором тестировщики учатся тестировать. Фронтенд — SPA (одностраничное приложение), десктоп-first (адаптив под мобильные не требуется). Бэкенд намеренно содержит «учебные баги» — их НЕ нужно чинить или маскировать в UI: тестировщикам должно быть что находить.

**Роли пользователей — три, каждая приходит одной точной строкой:**

| Роль | Строка | Что может |
|------|--------|-----------|
| Обычный пользователь | `ROLE_USER` | Счета, депозиты, переводы, история |
| Администратор | `ROLE_ADMIN` | Управление пользователями; банковские операции ему ЗАПРЕЩЕНЫ (получает 403) |
| Кредитный пользователь | `ROLE_CREDIT_SECRET` | Всё, что ROLE_USER, плюс кредиты |

Важно для UI: иерархия ролей существует только на бэкенде. Фронт сравнивает строку роли ТОЧНО (`===`). Пользователь имеет ровно одну роль.

**Дефолтный пользователь (создаётся автоматически):** логин `admin`, пароль `123456`, роль `ROLE_ADMIN`. Других пользователей в свежей системе НЕТ — их создаёт админ.

## 2. Аутентификация

### POST /api/auth/token/login (публичная, единственная без токена)

Запрос: `Content-Type: application/json`

```json
{"username": "admin", "password": "123456"}
```

Успех — HTTP 200:

```json
{"token": "<JWT>", "user": {"username": "admin", "role": "ROLE_ADMIN"}}
```

- `user.role` — ОДНА строка (см. таблицу ролей). Разбирать JWT на клиенте не нужно.
- Токен передаётся во все остальные ручки заголовком `Authorization: Bearer <JWT>`.

Ошибки:

| Случай | HTTP | Живое тело |
|--------|------|------------|
| Неверный логин или пароль | 401 | `{"error":"Invalid credentials"}` |
| Пустое тело / нет полей | 400 | `{"error":"Missing username or password"}` |

**Время жизни токена (снято с живого токена): 259200 секунд = 72 часа.**
Ручки обновления (refresh) НЕ существует: когда токен истёк, любая ручка вернёт HTTP 401 `{"code":401,"message":"Expired JWT Token"}` — пользователя нужно отправить на повторный логин. В макетах стоит предусмотреть состояние «сессия истекла — войдите заново».

Виды 401 (важно, их несколько и они разные):

| Тело | Что означает | Действие фронта |
|------|--------------|-----------------|
| `{"code":401,"message":"Expired JWT Token" / "Invalid JWT Token" / "JWT Token not found"}` | Токен истёк/битый/отсутствует | Разлогинить, редирект на логин |
| `{"error":"Invalid credentials"}` | Неверные креды на форме логина | Показать у формы «Неверный логин или пароль» (единственный переписанный текст бэкенда) |
| `{"error":"Forbidden: Admin access required"}` | НЕ-админ дёрнул админскую ручку (токен живой!) | НЕ разлогинивать, показать «нет прав» |
| `{"error":"Unauthorized"}` | Запрос к админской ручке без токена | Показать ошибку; авто-разлогина и редиректа нет (D-06) — до страниц без токена гварды не допускают |

## 3. Все 12 ручек

Формат каждой: метод, URL, доступ, тело запроса, живой успешный ответ, живые ошибки.

### 3.1 POST /api/auth/token/login — вход

Описана в разделе 2.

### 3.2 POST /api/account/create — открыть счёт

- Доступ: любой залогиненный, КРОМЕ админа. Тело запроса не требуется.
- Успех — HTTP 201:

```json
{"id": 1, "number": "3170988", "balance": 0}
```

`id` — внутренний идентификатор (им оперируют все остальные ручки!), `number` — 7-значный номер счёта СТРОКОЙ (декоративный: ни одна ручка не принимает его на вход), `balance` — число.

| Ошибка | HTTP | Живое тело |
|--------|------|------------|
| Уже 2 счёта | 409 | `{"error":"User already has maximum number of accounts(2)"}` |
| Под админом | 403 | `{"error":"Admins cannot create bank accounts"}` |

### 3.3 POST /api/account/deposit — пополнить счёт

- Доступ: владелец счёта (не админ).
- Запрос:

```json
{"accountId": 1, "amount": 1500}
```

- Успех — HTTP 200:

```json
{"id": 1, "balance": 1500}
```

| Ошибка | HTTP | Живое тело |
|--------|------|------------|
| Сумма вне лимита 1000–9000 | 400 | `{"error":"Amount must be between 1000 and 9000"}` |
| Пустое тело (несколько нарушений склеиваются через `\n`) | 400 | `{"error":"Account accountId is required\nAmount is required"}` |
| Чужой или несуществующий счёт | 404 | `{"error":"Account 3 not found or does not belong to userId 2"}` |
| Под админом | 403 | `{"error":"Admins cannot deposit"}` |

### 3.4 POST /api/account/transfer — перевод

- Доступ: владелец счёта-отправителя (не админ). Счёт-получатель — ЛЮБОЙ существующий (свой или чужой).
- Запрос (⚠️ оба счёта — по внутреннему `id`, НЕ по 7-значному номеру):

```json
{"fromAccountId": 1, "toAccountId": 2, "amount": 500}
```

- Успех — HTTP 200 (баланса получателя в ответе НЕТ):

```json
{"fromAccountId": 1, "toAccountId": 2, "fromAccountIdBalance": 1000}
```

| Ошибка | HTTP | Живое тело |
|--------|------|------------|
| Перевод самому себе (тот же счёт) | 400 | `{"error":"Cannot transfer to the same account"}` |
| Сумма вне лимита 500–10000 | 400 | `{"error":"Amount must be between 500 and 10000"}` |
| Недостаточно средств | 422 | `{"error":"Insufficient funds. Current balance: 1000.00, required: 9999.00"}` |
| Получатель не существует | 404 | `{"error":"Account with ID 99 not found"}` |
| Счёт списания чужой | 404 | `{"error":"Account 1 not found or does not belong to userId 3"}` |
| Под админом (текст про deposit — учебный баг) | 403 | `{"error":"Admins cannot deposit"}` |

### 3.5 GET /api/account/transactions/{id} — история операций счёта

- Доступ: только владелец счёта. `{id}` — внутренний id счёта.
- Успех — HTTP 200 (живое тело):

```json
{
  "id": 1,
  "number": "3170988",
  "balance": 1000,
  "transactions": [
    {"transactionId": 2, "type": "transfer_out", "amount": -500, "fromAccountId": 1, "toAccountId": 2, "createdAt": "2026-07-04 04:31:42", "creditId": null},
    {"transactionId": 1, "type": "deposit", "amount": 1500, "fromAccountId": null, "toAccountId": 1, "createdAt": "2026-07-04 04:31:08", "creditId": null}
  ]
}
```

- `type`: `deposit` | `transfer_in` | `transfer_out` | `credit_issuance` | `credit_repayment`
- `amount` — число СО ЗНАКОМ: исходящие операции отрицательные, входящие положительные
- Сортировка: новые сверху. Пагинации НЕТ — приходит всё сразу
- `creditId` заполняется у кредитных операций, иначе `null`; `fromAccountId` у депозита — `null`

| Ошибка | HTTP | Живое тело |
|--------|------|------------|
| id ≤ 0 | 400 | `{"error":"Invalid account ID format"}` |
| Чужой/несуществующий счёт | 404 | `{"error":"Account 3 not found or does not belong to userId 2"}` |

### 3.6 POST /api/credit/request — оформить кредит

- Доступ: ТОЛЬКО `ROLE_CREDIT_SECRET`.
- Запрос (счёт — свой, по внутреннему id):

```json
{"accountId": 3, "amount": 12000, "termMonths": 12}
```

- Успех — HTTP 201 (⚠️ поле называется `id` — это id счёта; `balance` — новый баланс счёта, кредит зачислен на него):

```json
{"id": 3, "amount": 12000, "termMonths": 12, "balance": 12000, "creditId": 1}
```

| Ошибка | HTTP | Живое тело |
|--------|------|------------|
| Уже есть активный кредит (⚠️ статус 404, не 409!) | 404 | `{"error":"Only one active credit allowed per user"}` |
| Пользователь без кредитной роли | 403 | ⚠️ HTML-страница (~159 КБ, не JSON!) с заголовком «Forbidden: ROLE_CREDIT access required (403 Forbidden)» |
| Сумма/срок вне лимитов (5000–15000, 1–60 мес) | 400 | `{"error":"..."}` — текст нарушения |
| Чужой счёт | 404 | `{"error":"..."}` |

⚠️ Почти все бизнес-ошибки кредитных ручек приходят со статусом **404** независимо от причины — учебный баг.

### 3.7 POST /api/credit/repay — погасить кредит

- Доступ: ТОЛЬКО `ROLE_CREDIT_SECRET`. Погашение списывается со счёта.
- Запрос (все ТРИ поля обязательны):

```json
{"creditId": 1, "accountId": 3, "amount": 12000}
```

- Успех — HTTP 200:

```json
{"creditId": 1, "amountDeposited": 12000}
```

**Правило: кредит гасится строго ОДИН раз и строго ТОЧНОЙ полной суммой долга.** Частичное погашение невозможно.

| Ошибка | HTTP | Живое тело |
|--------|------|------------|
| Сумма меньше долга | 422 | `{"error":"The amount is not enough. Credit balance: -12000"}` |
| Сумма больше долга | 422 | `{"error":"Repayment amount exceeds remaining debt"}` |
| На счёте не хватает денег | 422 | `{"error":"Insufficient funds. Current balance: 12000.00, required: 15000.00"}` |
| Кредит уже погашен | 409 | `{"error":"The credit has already been repaid"}` |
| Кредит не найден / не этого пользователя | 404 | `{"error":"Credit with ID 77 was not found or does not belong to the user"}` |

Нюанс порядка проверок: проверка «хватает ли денег на счёте» срабатывает РАНЬШЕ проверки «кредит уже погашен». Повторное погашение при пустом счёте вернёт 422 про Insufficient funds, а не 409.

### 3.8 GET /api/credit/history — история кредитов

- Доступ: ТОЛЬКО `ROLE_CREDIT_SECRET`. Без параметров.
- Успех — HTTP 200 (живое тело: один активный кредит и один погашенный):

```json
{
  "userId": 3,
  "credits": [
    {"creditId": 2, "accountId": 3, "amount": 5000, "termMonths": 6, "balance": -5000, "createdAt": "2026-07-04 04:32:50"},
    {"creditId": 1, "accountId": 3, "amount": 12000, "termMonths": 12, "balance": 0, "createdAt": "2026-07-04 04:32:10"}
  ]
}
```

- **Долг хранится ОТРИЦАТЕЛЬНЫМ числом**: `balance: -5000` = активный кредит с долгом 5000; `balance: 0` = погашен. В UI долг показывать положительным числом.
- Новые кредиты сверху.

### 3.9 POST /api/admin/create — создать пользователя

- Доступ: ТОЛЬКО `ROLE_ADMIN`.
- Запрос:

```json
{"username": "user1", "password": "User1Pass!", "role": "ROLE_USER"}
```

- `role`: одна из `ROLE_USER`, `ROLE_ADMIN`, `ROLE_CREDIT_SECRET`
- `username`: латиница и цифры, 3–15 символов, без спецсимволов
- `password`: от 8 символов, минимум 1 заглавная, 1 строчная, 1 цифра, 1 спецсимвол

Успех — ⚠️ HTTP **200** (не 201), и в ответе — **хэш пароля** (учебная утечка, в UI не показывать):

```json
{"id": 2, "username": "user1", "password": "<bcrypt-hash>", "role": "ROLE_USER"}
```

| Ошибка | HTTP | Живое тело |
|--------|------|------------|
| Дубликат username (⚠️ ЧУЖОЙ текст про счета — учебный баг) | 409 | `{"error":"User already has maximum number of accounts(2)"}` |
| Слабый пароль (нарушения склеены через `\n`) | 400 | `{"error":"Password must be at least 8 characters long\nPassword must contain at least one uppercase letter\nPassword must contain at least one number\nPassword must contain at least one special character"}` |
| Несуществующая роль (⚠️ текст умалчивает про ROLE_CREDIT_SECRET, но она ПРИНИМАЕТСЯ) | 400 | `{"error":"The role must be: ROLE_USER or ROLE_ADMIN"}` |

### 3.10 GET /api/admin/users — список всех пользователей

- Доступ: ТОЛЬКО `ROLE_ADMIN`. Без параметров.
- Успех — HTTP 200 (сортировка по id по возрастанию):

```json
[
  {"id": 1, "username": "admin", "role": "ROLE_ADMIN"},
  {"id": 2, "username": "user1", "role": "ROLE_USER"},
  {"id": 3, "username": "credit1", "role": "ROLE_CREDIT_SECRET"}
]
```

⚠️ Квирк: если эту (или любую админскую) ручку дёрнет НЕ-админ с живым токеном, придёт HTTP **401** (не 403) с телом `{"error":"Forbidden: Admin access required"}`. Пользователя НЕ нужно разлогинивать — его токен валиден.

### 3.11 DELETE /api/admin/users/{id} — удалить пользователя

- Доступ: ТОЛЬКО `ROLE_ADMIN`. `{id}` — id пользователя.
- Успех — HTTP 200 (удаление «мягкое»: счета, история и кредиты пользователя остаются в базе):

```json
{"message": "User deleted successfully"}
```

| Ошибка | HTTP | Живое тело |
|--------|------|------------|
| Админ удаляет сам себя | 400 | `{"error":"Cannot delete your own account"}` |
| Пользователь не найден | 404 | `{"error":"User not found"}` |

### 3.12 DELETE /api/admin/users — удалить ВСЕХ пользователей

- Доступ: ТОЛЬКО `ROLE_ADMIN`. Удаляет всех, кроме текущего админа.
- Успех — HTTP 200: `{"message": "All users except current admin deleted successfully", "deleted_count": N}`
- В обязательных требованиях UI этой ручки нет; полезна тестировщикам как «сброс». Если включать в макет — только с жёстким подтверждением (операция массовая).

## 4. Лимиты и механики

| Операция | Лимит |
|----------|-------|
| Депозит | 1000–9000 за раз |
| Перевод | 500–10000 за раз |
| Кредит | сумма 5000–15000, срок 1–60 месяцев |
| Счета | максимум 2 на пользователя |
| Кредиты | максимум 1 АКТИВНЫЙ на пользователя (после полного погашения можно взять новый) |

Механики:

- Кредит зачисляется на выбранный счёт обычным пополнением; долг хранится отрицательным `balance` кредита.
- Погашение — строго один раз, строго точной полной суммой, деньги списываются со счёта (на счёте их должно хватать).
- История транзакций: `amount` со знаком (исходящие отрицательные), новые записи сверху, пагинации нет.
- Валидация на бэкенде строгая; несколько нарушений в одном запросе приходят одной строкой, склеенные через `\n` — UI стоит показывать их списком.
- Ошибки бэкенда предназначены тестировщикам — показывать их тексты дословно, не переписывать.

## 5. Аномалии и учебные баги (НЕ чинить, НЕ маскировать)

1. **Ручки списка счетов пользователя НЕ СУЩЕСТВУЕТ.** Фронт узнаёт счета только из ответов создания счёта и истории транзакций (по известному id). После чистого логина (новый браузер) список счетов взять НЕОТКУДА — см. флаг для дизайнера в разделе 6.
2. **Перевод принимает внутренний `toAccountId` (id счёта), а НЕ 7-значный номер.** Ручки поиска счёта по номеру нет. Поле «номер счёта получателя» в макете обязано означать id.
3. **Почти все бизнес-ошибки кредитных ручек приходят как 404** — включая «Only one active credit allowed per user». По статусу причину не понять — только по тексту.
4. **403 кредитных ручек для пользователей без роли — HTML-страница** (~159 КБ), не JSON. Единственный не-JSON ответ API.
5. **Создание пользователя возвращает 200 (не 201) и содержит хэш пароля** — утечка. В UI поле `password` из ответа не отображать.
6. **Дубликат username → 409 с чужим текстом** `User already has maximum number of accounts(2)` — текст про счета, хотя проблема в имени.
7. **Не-админ на админских ручках получает HTTP 401 (не 403)** с телом `{"error":"Forbidden: Admin access required"}` — токен при этом живой, разлогинивать нельзя.
8. **Текст ошибки валидации роли врёт**: говорит «must be: ROLE_USER or ROLE_ADMIN», но `ROLE_CREDIT_SECRET` реально принимается.
9. **Ошибка «Admins cannot deposit» на переводе** — копипаст-текст: админ при переводе получает сообщение про депозит.
10. **Повторное погашение кредита при пустом счёте даёт 422 (Insufficient funds), а не 409** — проверка средств идёт раньше проверки «уже погашен».
11. **Встроенная OpenAPI-документация бэкенда врёт минимум в 4 местах** (лимиты депозита, имя поля в ответе кредита, обязательность полей погашения, допустимые роли). Истина — в этом документе.

## 6. Маппинг «экран → ручки»

| Экран | Ручки | Источник данных |
|-------|-------|-----------------|
| **Логин** | POST /api/auth/token/login | Форма username/password; после успеха редирект по роли: ROLE_ADMIN → админка, остальные → кабинет |
| **Админка: таблица пользователей** | GET /api/admin/users | Массив `{id, username, role}` |
| **Админка: создание пользователя** | POST /api/admin/create | Форма username/password + выбор роли (дропдаун/радио из трёх ролей); правила пароля и имени показать рядом с полями |
| **Админка: удаление пользователя** | DELETE /api/admin/users/{id} | Кнопка в строке таблицы + подтверждение |
| **Кабинет: Счета** | POST /api/account/create, POST /api/account/deposit | Карточки счетов: номер + баланс; кнопка «Открыть счёт» (дизейбл при 2 счетах); пополнение с выбором счёта и подсказкой лимита 1000–9000 |
| **Кабинет: Переводы** | POST /api/account/transfer | Два режима радиобаттоном: «между своими» (оба счёта — дропдауны) и «другому пользователю» (получатель — ввод id счёта); подсказка лимита 500–10000 |
| **Кабинет: История** | GET /api/account/transactions/{id} | Выбор счёта → таблица: тип, сумма со знаком, дата, контрагент |
| **Кабинет: Кредиты** (виден только `ROLE_CREDIT_SECRET`) | POST /api/credit/request, POST /api/credit/repay, GET /api/credit/history | Оформление в модалке (сумма 5000–15000, срок 1–60 мес, выбор счёта, чекбокс согласия); карточка активного кредита (долг положительным числом) с кнопкой «Погасить целиком»; история кредитов |

**⚠️ Явный флаг для дизайнера:** после чистого логина список счетов взять НЕОТКУДА (аномалия №1). В макете нужно предусмотреть одно из двух: (а) фронт ведёт собственный реестр известных счетов (пополняется при создании счёта и просмотре операций), и/или (б) поле ручного ввода id счёта («добавить счёт по id»). Пустое состояние «счета неизвестны» обязательно.

Также: у админа НЕТ банковских разделов (счета/переводы/кредиты дают ему 403) — его навигация состоит только из управления пользователями. У `ROLE_USER` нет раздела «Кредиты».

## 7. Чек-лист 9 обязательных UI-элементов

Эти элементы — критерий приёмки работы. Каждый обязан присутствовать в макетах:

| # | Элемент | Где в макете |

## 8. Сквозные пользовательские сценарии

### Сценарий А: от админа до перевода

1. Админ логинится (`admin`/`123456`) → попадает в админку.
2. Админ создаёт пользователя `user1` с ролью `ROLE_USER` → тот появляется в таблице.
3. `user1` логинится → попадает в кабинет (пустое состояние: счетов нет).
4. Открывает счёт («Счёт создан успешно!») → карточка счёта с номером и балансом 0.
5. Пополняет счёт на 1500 → алерт успеха, баланс 1500 без перезагрузки страницы.
6. Открывает второй счёт, переводит 500 между своими (радиобаттон «между своими») → алерт успеха, балансы обновились.
7. Открывает историю → таблица: перевод −500 (transfer_out), депозит +1500, новые сверху.

### Сценарий Б: кредитный флоу

1. Админ создаёт `credit1` с ролью `ROLE_CREDIT_SECRET`.
2. `credit1` логинится → в кабинете виден раздел «Кредиты» (у `user1` его нет).
3. Открывает счёт, жмёт «Оформить кредит» → модалка: сумма 5000–15000, срок 1–60 мес, выбор счёта дропдауном, чекбокс согласия (кнопка неактивна без него).
4. Оформляет кредит 12000 → алерт успеха; баланс счёта +12000; карточка активного кредита: долг 12000 (показан положительным), кнопка «Погасить целиком».
5. Пытается оформить второй кредит → ошибка «Only one active credit allowed per user» (пришла как 404 — показать текст).
6. Гасит кредит одной кнопкой точной суммой → алерт успеха; кредит в истории с долгом 0.

### Сценарий В: ошибочные пути (обязательны для полигона)

- Неверный пароль на логине → у формы «Неверный логин или пароль» (единственный переписанный текст бэкенда: живое тело `{"error":"Invalid credentials"}`); без редиректа.
- Депозит 999 → «Amount must be between 1000 and 9000» (форма не блокирует отправку — бэкенд должен получить запрос и ответить ошибкой).
- Перевод 15000 → «Amount must be between 500 and 10000»; перевод при нехватке средств → «Insufficient funds. Current balance: X, required: Y».
- Третий счёт → кнопка задизейблена; если запрос всё же ушёл — «User already has maximum number of accounts(2)».
- Погашение кредита неточной суммой → «The amount is not enough. Credit balance: -12000».
- Создание пользователя с занятым именем → «User already has maximum number of accounts(2)» (чужой текст — показать как есть).

Философия: валидация на фронте МЯГКАЯ — подсказки лимитов рядом с полями, но отправка не блокируется; тестировщики должны видеть живые ошибки бэкенда дословно.

## 9. Форматы данных

| Данные | Формат | Пример |
|--------|--------|--------|
| Деньги (балансы, суммы) | JSON number (целое или с дробной частью), НЕ строка | `1000`, `1500.75` |
| Сумма в истории операций | число СО ЗНАКОМ: исходящие отрицательные | `-500`, `1500` |
| Долг по кредиту | отрицательное число; `0` = погашен | `-5000` |
| Дата/время (`createdAt`) | строка `"YYYY-MM-DD HH:MM:SS"`, без миллисекунд и таймзоны | `"2026-07-04 04:31:42"` |
| Номер счёта (`number`) | строка из 7 цифр (декоративная — на вход ручки её не принимают) | `"3170988"` |
| Идентификаторы (`id`, `accountId`, `creditId`, `transactionId`) | целые числа | `1`, `42` |
| Роль | одна строка из трёх значений | `"ROLE_CREDIT_SECRET"` |
| Тип операции | строка | `"deposit"`, `"transfer_in"`, `"transfer_out"`, `"credit_issuance"`, `"credit_repayment"` |
| Ошибки | `{"error": "текст"}`; несколько нарушений валидации — одной строкой через `\n`; истёкший токен — `{"code":401,"message":"..."}` | см. раздел 2 и 3 |

Суммы в текстах ошибок бэкенд форматирует с двумя знаками после точки: `Current balance: 1000.00`.
